# Internationalization and Translation Workflow

SparkyFitness web uses `i18next`, `react-i18next`, the browser language detector, and the HTTP backend. Arabic is the first-time default, English is the guaranteed fallback, and an explicit saved user preference always wins.

## Source of Truth

Translation ownership is intentionally split:

- `SparkyFitnessFrontend/public/locales/en/translation.json` is the editable source catalog in this repository.
- Non-English catalogs, including Arabic, are edited in `CodeWithCJ/SparkyFitnessTranslations` through Weblate or a reviewed source change.
- `.github/workflows/sync-translations.yml` copies English to the translations repository and copies reviewed non-English catalogs back here.

Do not hand-edit a non-English catalog in the main repository. The next translation sync will overwrite it and may create conflicts.

## Runtime Setup

The main files are:

- `SparkyFitnessFrontend/src/i18n.ts` — supported languages, detection order, catalog fallback, and HTTP load path.
- `SparkyFitnessFrontend/src/utils/localePolicy.ts` — Arabic default, language storage key, direction, presentation locale, and Saudi defaults.
- `SparkyFitnessFrontend/src/components/LanguageHandler.tsx` — synchronizes `i18next` and the document's `lang`/`dir` attributes with the saved preference.
- `SparkyFitnessFrontend/src/contexts/PreferencesContext.tsx` — persists the user's explicit language and other presentation preferences.

Catalogs are loaded at runtime from:

```text
/locales/{{lng}}/{{ns}}.json
```

The default namespace is `translation`.

## Adding or Rewriting UI Copy

1. Add or revise the English source message in `public/locales/en/translation.json`.
2. Render it with `useTranslation()` and a stable semantic key.
3. Preserve interpolation variables exactly across languages.
4. Update the Arabic message in the translations source repository, using the Saudi/Najdi voice rules in `agent-docs/najdi-arabic-localization-spec.md`.
5. Run the catalog audit and the closest component tests.
6. Sync reviewed non-English catalogs through the translation workflow.

Example:

```tsx
import { useTranslation } from "react-i18next";

export function EmptyDiary() {
  const { t } = useTranslation();

  return <p>{t("diary.empty.description", "No entries yet for this day.")}</p>;
}
```

Avoid:

- hardcoded visible English in components;
- sentence fragments joined in code;
- using an English sentence as the translation key;
- changing or translating user-entered content, provider names, URLs, or API keys;
- adding fallback copy that is less polished than the catalog message.

## Saudi Arabic and RTL

- Arabic uses `ar-SA` presentation metadata and RTL document direction.
- Health-record day strings stay Gregorian `YYYY-MM-DD` values at data boundaries.
- UI dates default to `dd/MM/yyyy`; metric units and kcal remain the Saudi defaults.
- Use logical start/end CSS properties. Decide whether directional icons, charts, timelines, and media controls should mirror based on meaning, not page direction alone.
- Use light Najdi warmth only in conversational or motivational copy. Medical, pregnancy, medication, privacy, security, and destructive-action copy stays precise and professional.
- English remains the fallback while a catalog is incomplete so users never see raw message keys.

## Catalog Audit

From `SparkyFitnessFrontend/`:

```bash
pnpm run test:i18n
pnpm run audit:i18n -- ar
```

`test:i18n` verifies the audit logic. `audit:i18n` compares a target catalog to English and fails when keys are missing or interpolation variables differ. It reports:

- source and translated message counts;
- coverage percentage;
- missing keys;
- extra/stale keys;
- placeholder mismatches such as `{{name}}` becoming `{{user}}`.

The audit's non-zero exit is expected while a catalog is incomplete. Do not weaken the check to make an incomplete catalog appear healthy.

## Adding a Supported Language

1. Add the language code and native display name in `src/utils/languageUtils.ts`.
2. Create the language catalog in the translations source repository.
3. Reach full key and interpolation parity with English.
4. Verify language selection, fallback, `lang`, and `dir` behavior.
5. Test the critical journeys at mobile, tablet, and desktop widths.
6. Run the sync workflow to bring the reviewed catalog into the main repository.

RTL languages also require a semantic layout and icon-mirroring review before release.
